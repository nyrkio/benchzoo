benchzoo / vegeta sample target

This page is a static HTTP target served by a minimal nginx instance on port 8080. It exists solely to give vegeta a stable endpoint to attack while benchzoo captures vegeta's JSON and text output for parser development.

Why this page exists

Unlike synthetic benchmarks that time a sleep or a CPU loop, vegeta fundamentally measures real HTTP request/response latency at a configured attack rate. It needs something to talk to. Rather than pointing vegeta at an external service (unstable, rate- limited, noisy), we stand up a minimal nginx instance serving this page, hit it locally over the loopback interface, and record vegeta's report.

What vegeta actually measures here

Latency percentiles (p50, p95, p99, max, min, mean), requests per second achieved versus requested rate, success ratio, status-code distribution, and bytes in/out — the fields vegeta emits in its JSON report. The measurements are a property of nginx + vegeta + the runner's kernel, not of this page's content. The page exists to be served, not to be scrutinized.

Why the page is not empty

vegeta's default target is a simple GET and the response needs a non-trivial body for the bytes_in number to be interesting. A few paragraphs of prose is enough to make that metric non-zero without inflating page size to the point where bandwidth dominates the latency measurement. The goal is a realistic-ish static page, not a stress test of nginx's sendfile path.

What this page is not

This is not a meaningful web application. There are no scripts, no external assets, no cookies, no forms. A real web-performance benchmark would look completely different (and Lighthouse, not vegeta, is the right tool for that). This is an HTTP load-test target, which is a different question entirely.
